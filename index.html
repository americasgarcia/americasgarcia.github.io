ABCU Course Advising System

Hash Map Sorting CSV Parsing Input Validation
The Problem

Design and implement a course planning tool for an academic advising program. The system needed to load a course catalog from a CSV file, validate prerequisites, sort courses alphanumerically, and allow users to look up any course and its dependencies in real time.

Architectural Decision

The core design question was data structure selection. After analyzing runtime complexity tradeoffs between vectors, binary search trees, and hash maps, an unordered_map was chosen for O(1) average-case lookups by course number. Sorting was deferred to a separate pass using std::sort only when the user requested a full catalog listing, keeping the common lookup path fast.

typedef std::unordered_map<std::string, Course> Catalog; // O(1) lookup by course number bool loadCatalogFromCSV(const std::string& filename, Catalog& catalog) { // Validate prerequisites exist before accepting the row for (const auto& prereq : course.prereqs) { if (temp.find(prereq) == temp.end()) return false; } }
Key Engineering Challenges

File input validation was a significant challenge. Raw CSV data contained inconsistent whitespace, duplicate course entries, and missing prerequisite references. A strict two-pass validation approach was implemented: first collecting all course numbers, then verifying every prerequisite reference resolved to a known course before committing the catalog to memory. Invalid files were rejected entirely rather than partially loaded, preventing silent data corruption.

Outcome

The system handled malformed input gracefully, validated all prerequisite chains at load time, and delivered sorted catalog output with full prerequisite detail. The design demonstrated how upfront data structure selection dramatically affects application performance and maintainability.

02
Java  /  JUnit

Contact, Task & Appointment Services

Test-Driven Dev JUnit Requirements Tracing Input Validation
The Problem

Build three backend service modules for managing contacts, tasks, and appointments. Each module required strict data validation enforced at the object level, unique ID management, and full CRUD operations. Every requirement had to be verifiable through automated unit tests.

Design Philosophy

Requirements were treated as executable specifications. Before writing a single line of service logic, each user requirement (unique IDs, character limits, null rejection, immutability constraints) was translated directly into a JUnit test case. This forced clarity on exactly what the code needed to do before it was written, and ensured no requirement could be silently dropped.

// Requirements enforced at construction — not as an afterthought public Contact(String contactId, String firstName, ...) { if (contactId == null || contactId.length() > 10) throw new IllegalArgumentException("bad id"); this.contactId = contactId; // immutable after construction setFirstName(firstName); // validated via setter }
Requirements Traceability

Each test method was mapped explicitly to a requirement: unique ID enforcement, null rejection, field length constraints, and immutability of the contact ID post-creation. The test suite served as living documentation, making it immediately clear what the system guaranteed and where it would fail intentionally versus unexpectedly.

Outcome

Full test coverage across all three service modules, with every requirement traceable to a specific test case. The project demonstrated how test-driven development is not just a quality practice but a requirements management discipline that keeps implementation honest.

03
Python  /  MongoDB  /  Plotly Dash

Grazioso Salvare Animal Rescue Dashboard

CRUD Module MongoDB Plotly + Dash Leaflet Maps Modular Design
The Problem

Build a full-stack interactive dashboard for Grazioso Salvare, an animal rescue organization, to help staff identify dogs suitable for specific rescue training programs. The system connected a live MongoDB database to a multi-component Jupyter Dash dashboard with real-time filtering, a data table, a Plotly pie chart, and a Leaflet map all synchronized via callbacks.

Architecture: Two-Layer Design

A standalone AnimalShelter CRUD class encapsulated all MongoDB interactions. The dashboard imported that module and focused entirely on user interaction and visualization. Neither layer knew the internals of the other, so any database change only required updating one class.
